# Matrix System Roadmap

日期：2026-03-23

## 背景

参考产品：

- [矩阵通 - 新榜旗下多平台新媒体矩阵一站式管理系统](https://matrix.newrank.cn/)
- [关于矩阵通](https://matrix.newrank.cn/about)
- [什么是矩阵通 PDF](https://matrixcdn.newrank.cn/matrix/pdf/1%E5%88%86%E9%92%9F%E5%BF%AB%E9%80%9F%E4%BA%86%E8%A7%A3%E7%9F%A9%E9%98%B5%E9%80%9A.pdf)

根据公开资料，矩阵通的能力不是单一“发帖工具”，而是更完整的矩阵管理系统，覆盖：

- 多平台账号集中管理
- 数据监测与运营评估
- 任务下发与结果统计
- 风险检测
- 竞品对比
- 内容资产留存

## 当前项目所处阶段

当前仓库已经具备的是：

### 1. 发布执行层

由 `social-push` 提供：

- 多平台 workflow
- 浏览器自动化发布
- 实测结果沉淀

### 2. 账号治理层

由新建的 `account-matrix` 提供：

- 账号矩阵模板
- 验证矩阵模板
- 发布前检查清单

也就是说，当前项目已经覆盖了矩阵系统里最底层的两块：

- **执行**
- **台账**

但还没有做成完整的“矩阵管理平台”。

## 推荐演进阶段

### Phase 1：模板矩阵

目标：

- 先把“谁在用哪个号、哪个形态验证到哪一步”记清楚

当前对应：

- `account-matrix/`

关键产物：

- `account-matrix/templates/account-matrix.md`
- `account-matrix/templates/verification-matrix.md`
- `account-matrix/templates/preflight-checklist.md`

### Phase 2：可执行矩阵

目标：

- 在发布前自动读取矩阵
- 自动做账号确认与验证状态检查

建议能力：

- 根据 `platform + account_alias` 输出预检结果
- 检查当前页面显示名是否匹配矩阵
- 检查当前 `content_type` 是否已真实验证

这一步仍然不需要大而重的后台系统。

### Phase 3：团队矩阵

目标：

- 支持一人多号、多人多号、平台多组管理

建议能力：

- 账号分组
- 默认账号策略
- 团队别名
- 平台 owner
- 使用范围，如“只测图帖”“品牌号”“主号”

### Phase 4：运营矩阵

目标：

- 从“账号台账”升级到“矩阵运营”

建议能力：

- 发布任务清单
- 结果回填
- 验证状态自动更新
- 风险记录
- 异常原因归档

### Phase 5：数据矩阵

目标：

- 补齐接近矩阵通的核心差异化能力

建议能力：

- 多平台账号指标采集
- 发文统计
- 增长趋势
- 内容表现
- 风险告警

这一阶段才真正接近矩阵通所代表的“管理系统”。

## 对当前仓库的建议

不要一步到位照着矩阵通做“大系统”。

当前最合理的路径是：

1. 先把 `account-matrix` 用起来
2. 再做“矩阵读取 + 发布前校验”
3. 再做“结果回填”
4. 最后再考虑统一数据面板

原因很简单：

- 当前已经有真实可用的发布执行层
- 最缺的是“账号与验证状态的结构化治理”
- 这一步成本低，但收益非常高

## 一句话总结

矩阵通是“发布 + 账号治理 + 数据监测 + 组织管理”的完整矩阵平台。

当前仓库已经完成了：

- `发布执行层`
- `账号治理模板层`

下一步不该直接冲全量平台，而应该先把：

- **矩阵模板 -> 可执行预检 -> 结果回填**

这条链路做通。
